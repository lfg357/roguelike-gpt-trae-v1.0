```
Decision ID：DEC-001
标题：角色与 VFX 美术从静态图集改为逐帧动作序列
状态：Implemented
日期：2026-07-13
决策人：Human + AI

背景：早期美术实现使用静态人物图和几张特效贴图叠加，不符合附录 G 要求的 6-12 帧逐帧动作和独立时间特效序列。静态叠加方式无法表达完整动作语义，且容易出现"图片轮播"感。

候选方案：
  A：保持静态图集，通过代码插值、缩放、旋转模拟动画
  B：制作逐帧动作序列，每个动作/特效为独立时间行，运行时按帧采样

选择方案：B

理由：
- 附录 G 明确要求角色动作组和 6-12 帧，特效为独立时间序列
- 静态叠加无法表达起势、峰值、回收的完整动作节奏
- 斜俯视角色旋转后透视会崩坏，不能靠旋转冒充朝向
- 逐帧序列虽然制作成本更高，但视觉质量和语义完整性有本质提升
- 长期来看，逐帧资源是动画系统的可靠基础，便于后续扩展新动作

影响：
- 优点：动作语义清晰、符合附录 G、视觉质量大幅提升、便于后续扩展
- 缺点：资源制作成本高、图集管理复杂度增加、需要处理脚点锚点和帧边界问题
- 后续工作：继续补全水墨粒子、顿帧叠加层、伤害数字等 P0 美术缺口
- 影响范围：角色渲染、VFX 系统、资源目录结构

关联文档：docs/ANIMATION_RENDERING.md、docs/APPENDIX_G_ART_ACCEPTANCE.md、docs/P0_ALIGNMENT.md
```

```
Decision ID：DEC-002
标题：角色朝向使用 X 轴镜像，不旋转斜俯视贴图
状态：Implemented
日期：2026-07-13
决策人：Human + AI

背景：斜俯视透视的角色贴图如果整体旋转，透视关系会崩坏（头顶和脚底比例错乱）。需要一种正确的左右朝向切换方案。

候选方案：
  A：旋转整个角色 Sprite
  B：按 X 轴镜像翻转（scale.x = -1）
  C：制作左右两套完整动作帧

选择方案：B

理由：
- 斜俯视角色旋转后透视关系完全错误，A 方案不可行
- C 方案虽然最完美但资源量翻倍，P0 阶段成本过高
- X 轴镜像在斜俯视角度下视觉可接受，且实现简单、性能好
- 行业内俯视角游戏普遍采用镜像方案

影响：
- 优点：实现简单、性能好、资源量不增加
- 缺点：严格来说左右不对称的细节会镜像，但俯视角下不明显
- 影响范围：角色渲染、VFX 方向型特效（特效仍可旋转）

关联文档：HANDOFF.md §5.1、docs/ANIMATION_RENDERING.md §3
```

```
Decision ID：DEC-003
标题：五行高级格局驱逐低级格局
状态：Implemented
日期：2026-07-12
决策人：Human + AI

背景：早期实现中五行所有子组合同时触发，导致 3 元素格局成立时，其包含的 2 元素和单元素格局也全部生效，效果叠加过于强势，且不符合设计文档的"格局层内部选一个主导格局"规则。

候选方案：
  A：所有格局同时生效，简单叠加
  B：按"精确签名 > 等阶 > 元素覆盖 > 优先级"选主导格局，高阶驱逐低阶
  C：玩家手动选择激活哪个格局

选择方案：B

理由：
- 设计文档明确要求格局层内部选一个主导格局
- 全部叠加会导致高阶格局与低阶没有质的区别，构筑深度不足
- 玩家手动选择操作复杂，且违背 Roguelike 自动构筑的流畅感
- B 方案既能体现高阶格局的价值，又能保持系统的自动性

影响：
- 优点：符合设计文档、构筑有深度、高阶格局有存在感
- 缺点：实现复杂度略高，需要正确计算格局层级
- 影响范围：五行规则系统、遗物平衡性

关联文档：scripts/FiveElementRules.gd、《设计文档 v4.0》附录 D
```

```
Decision ID：DEC-004
标题：五行圆满定位为均衡小幅提升，而非全属性尖峰
状态：Implemented
日期：2026-07-12
决策人：Human + AI

背景：早期实现将五行圆满做成大幅全属性提升，导致其成为唯一最优解，破坏了构筑多样性。设计文档中五行圆满的定位是"均衡、稳定、每项都有一点但都不极强"。

候选方案：
  A：大幅全属性尖峰，凑齐就是最强
  B：均衡小幅提升，每项都有但都不极端
  C：根据当前层数动态调整强度

选择方案：B

理由：
- 设计文档明确五行圆满是均衡稳定方向，不是最强 build
- 如果五行圆满太强，玩家会一味追求凑齐五行，丧失构筑多样性
- 均衡定位使得五行圆满作为"兜底选择"有价值，但不压制其他方向
- 符合 Roguelike 中"全能型通常不是最强"的设计常识

影响：
- 优点：构筑多样性更好、符合设计文档、各五行方向都有价值
- 缺点：五行圆满的"爽感"可能略弱
- 影响范围：五行规则系统、平衡性

关联文档：scripts/FiveElementRules.gd、《设计文档 v4.0》附录 D
```

```
Decision ID：DEC-005
标题：武器掉落采用新旧抉择面板，不自动覆盖
状态：Implemented
日期：2026-07-11
决策人：Human + AI

背景：武器掉落时如果直接替换旧武器，玩家可能会在不了解新武器详情的情况下被迫接受，且丧失策略选择感。

候选方案：
  A：自动替换旧武器，简单直接
  B：弹出抉择面板，显示完整详情，玩家选择替换/保留
  C：自动放入背包，玩家自行装备

选择方案：B

理由：
- 设计文档明确要求武器抉择机制
- Roguelike 中武器是核心道具，替换应该是有意识的决策
- 完整的武器详情展示帮助玩家做出有意义的选择
- A 方案剥夺玩家选择权，C 方案背包管理负担过重

影响：
- 优点：符合设计文档、玩家有策略感、信息透明
- 缺点：增加一次交互打断
- 影响范围：武器系统、UI 面板

关联文档：HANDOFF.md §4.3
```

```
Decision ID：DEC-007
标题：地图拓扑与房间门共用同一数据源
状态：Implemented
日期：2026-07-11
决策人：Human + AI

背景：早期实现为每个房间硬编码右侧门，导致地图图卷显示的连接和实际房间门的位置不一致，且商店没有返回出口。

候选方案：
  A：为每个房间类型硬编码固定门位置
  B：地图拓扑数据生成房间门，图卷和房间共用同一数据源
  C：房间根据相邻房间动态生成门，但与图卷数据独立维护

- 支持 8 方向连接，扩展性好

影响：
- 优点：数据一致、灵活、支持 8 方向连接
- 缺点：需要正确的拓扑数据结构
```

```
Decision ID：DEC-008
标题：动画过渡采用"单帧占优 + 极短能量融合"策略
状态：Implemented
日期：2026-07-13
决策人：Human + AI

背景：如果对整个人物帧做长时间交叉淡化，衣袖、刀身和身体轮廓会同时出现两份，残影反而更重。人物和能量特效需要不同的过渡策略。

候选方案：
  A：人物和 VFX 都用整帧交叉淡化
  B：人物单帧占优（硬切），VFX 不融合
  C：人物单帧占优 + 脚点锚点对齐，VFX 只在帧尾 26% 做短融合

选择方案：C

理由：
- 人物整帧融合会出现双头/双刀等严重残影，A 方案不可行
- B 方案太生硬，能量特效会有明显跳变感
- C 方案人物保持清晰轮廓，VFX 有柔和过渡，兼顾两者优点
- 帧尾 26% 的融合窗口很短（约 6.5ms），不会产生可辨识的双重贴图
- 两帧 alpha 之和恒为 1，不叠亮、不积累尾影

影响：
- 优点：人物清晰、VFX 柔和、无残影、无叠亮
- 缺点：实现复杂度较高，需要维护锚点元数据
- 影响范围：角色渲染、VFX 渲染系统

关联文档：docs/ANIMATION_RENDERING.md §3、§4
```

```
Decision ID：DEC-009
标题：项目采用 1280x720 固定设计分辨率 + keep 拉伸模式
状态：Implemented
日期：2026-07-14
决策人：Human + AI

背景：不同显示器宽高比不同，需要确定 UI 和游戏世界的缩放策略。早期曾使用 canvas_items 拉伸模式配合鼠标窗口坐标，导致 DPI 和留边影响输入映射。

候选方案：
  A：1280x720 视口 + canvas_items 拉伸（全屏拉伸）
  B：1280x720 视口 + keep 拉伸（等比缩放留边）
  C：自适应分辨率，UI 用锚点布局

选择方案：B

理由：
- keep 模式保证设计比例不变，不会出现 UI 拉伸变形
- 1280x720 是明确的设计目标，所有面板都在该尺寸下验证
- canvas_items 模式下鼠标坐标映射复杂，容易出 bug
- C 方案实现复杂度高，且难以保证所有分辨率下都无遮挡

影响：
- 优点：比例恒定、UI 布局可预测、输入映射简单
- 缺点：宽屏/窄屏会有黑边
- 影响范围：视口设置、鼠标输入、UI 布局

关联文档：docs/ANIMATION_RENDERING.md §6、project.godot
```

```
Decision ID：DEC-010
标题：运行时资源与绿幕源文件分目录存放
状态：Implemented
日期：2026-07-13
决策人：Human + AI

背景：美术资源有绿幕源文件（chroma key）和透明化后的运行时 PNG 两种状态。如果混放容易误将源文件导入 Godot，且目录语义不清晰。

候选方案：
  A：所有 PNG 混放，靠文件名区分
  B：source/ 目录放绿幕源文件（.gdignore 排除导入），animated/ 或 sequences/ 放运行时透明 PNG
  C：源文件放 art-resources/，运行时资源放 assets/

选择方案：B + C 结合

理由：
- source 目录用 .gdignore 排除 Godot 导入，从机制上避免误导入
- 运行时目录语义清晰，代码引用路径稳定
- art-resources/ 作为原始资源包归档，不直接参与运行时
- 符合"资源按语义和风格谨慎使用"的原则

影响：
- 优点：目录清晰、机制上防止误导入、源文件安全
- 缺点：新资源需要经过透明化流程才能进入运行时
- 影响范围：资源目录结构、资源制作流程

关联文档：HANDOFF.md §5.5、assets/characters/README.md、assets/vfx/README.md
```
