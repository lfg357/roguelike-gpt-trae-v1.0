# 2. 长期规则

```
Rule-001
名称：HANDOFF 优先原则
说明：新会话开始后，必须先读 HANDOFF.md，再读设计文档和代码。不得凭聊天摘要、旧截图或旧静态图集继续开发。
建立原因：防止 AI 基于过期上下文继续工作，导致偏离当前项目状态。
建立时间：2026-07-14
状态：Active
```

```
Rule-002
名称：设计文档权威原则
说明：严格按照《仙途·墨渊》设计文档 v4.0 推进。设计文档没有给出的正式配方、机制和美术语义，不要擅自当成定案。
建立原因：避免 AI 自行发明游戏内容，保持设计一致性。
建立时间：2026-07-14
状态：Active
```

```
Rule-003
名称：美术资源语义原则
说明：角色和特效必须按动作/时间批次制作。不能用几张静态图通过缩放、旋转和多层叠加冒充完整动画。
建立原因：早期美术实现不符合附录 G 要求，必须纠正。
建立时间：2026-07-14
状态：Active
```

```
Rule-004
名称：五行格局驱逐原则
说明：五行高级格局驱逐低级格局。同一批元素不能同时触发高阶格局及其包含的全部低阶格局。
建立原因：设计文档明确要求，且已在代码中实现。
建立时间：2026-07-14
状态：Active
```

```
Rule-005
名称：五行圆满均衡原则
说明：五行圆满是均衡、小幅、每项都有一点但都不极强，不是大幅全属性尖峰。
建立原因：设计文档明确要求，早期实现曾偏离。
建立时间：2026-07-14
状态：Active
```

```
Rule-006
名称：武器抉择原则
说明：武器掉落必须让玩家在新旧武器间选择，不能自动覆盖。
建立原因：设计文档明确要求。
建立时间：2026-07-14
状态：Active
```

```
Rule-007
名称：Boss 延迟结算原则
说明：Boss 死亡后先掉落战利品并生成中央漩涡门，玩家触发漩涡门后才弹结算。
建立原因：设计文档明确要求。
建立时间：2026-07-14
状态：Active
```

```
Rule-008
名称：地图拓扑统一原则
说明：地图连接方向与房间门必须来自同一拓扑数据。不能为每个房间硬编码右侧门，商店也必须有可返回出口。
建立原因：早期实现曾硬编码右门，导致地图与房间门不一致。
建立时间：2026-07-14
状态：Active
```

```
Rule-009
名称：角色朝向镜像原则
说明：角色使用统一的右向基准和斜俯视透视。运行时只允许按 X 轴镜像左右朝向，不得旋转角色贴图，否则透视会崩坏。
建立原因：斜俯视角色旋转后透视会出错。
建立时间：2026-07-14
状态：Active
```

```
Rule-010
名称：资源目录分离原则
说明：运行时透明 PNG 与绿幕源文件分目录存放。source 目录通过 .gdignore 排除 Godot 导入。不得直接加载绿幕源图。
建立原因：保持资源目录清晰，避免 Godot 导入源文件。
建立时间：2026-07-14
状态：Active
```

```
Rule-011
名称：1280x720 防遮挡原则
说明：所有 UI 面板和文字都必须在 1280x720 尺寸下实机检查，不得出现互相覆盖。
建立原因：UI 遮挡是反复出现的问题。
建立时间：2026-07-14
状态：Active
```

```
Rule-012
名称：新增内容测试原则
说明：任何新增开发内容（功能、数据、美术资源）必须在完成后及时安排对应测试。测试覆盖包括但不限于：冒烟测试回归、核心功能用例执行、实机视觉验收（如涉及美术）。禁止在未测试的情况下宣布完成。
建立原因：防止新增内容引入隐性bug，确保每次变更都经过验证。
建立时间：2026-07-14
状态：Active
```

```
Rule-013
名称：HANDOFF 更新原则
说明：每次工程变更（代码修改、文档创建、配置调整）完成后，必须及时更新 HANDOFF.md 记录变更内容。即使发生上下文压缩，恢复后也必须检查并补充缺失的记录。禁止因上下文压缩而忽略更新 HANDOFF.md。
建立原因：HANDOFF.md 是项目状态的唯一真实来源，必须保持实时更新以支持上下文恢复。
建立时间：2026-07-14
状态：Active
```

```
Rule-014
名称：代码审查原则
说明：所有代码修改完成后，必须执行代码审查流程：自检清单 → 语法检查 → 冒烟测试。审查不通过不得标记任务完成。审查规范详见 docs/CODE_REVIEW.md。
建立原因：2026-07-14 发生类型推断错误导致工程无法打开，暴露缺乏提交前检查机制的问题。必须建立严格的代码审查制度，防止类似问题重复出现。
建立时间：2026-07-14
状态：Active
```

```
Rule-015
名称：显式类型声明原则
说明：GDScript 中所有变量声明必须显式指定类型，禁止使用 := 依赖类型推断。比较表达式、三元表达式等复杂表达式的返回类型无法被正确推断，必须显式声明。
建立原因：2026-07-14 因使用 := 声明比较结果变量导致类型推断失败，工程无法编译。GDScript 类型推断对复杂表达式支持不完善，显式声明是唯一可靠方案。
建立时间：2026-07-14
状态：Active
```

# 3. 长期偏好

| 类别 | 偏好 |
|------|------|
| 输出风格 | 正式、简洁、直接 |
| 文档格式 | Markdown |
| 语言 | 中文 |
| 代码规范 | GDScript 官方风格，2 空格缩进 |
| 文件编码 | UTF-8 |
| 问题解决 | 先确认意图再实施，避免浪费时间 |
| 测试方式 | 优先 Godot F5 debug 模式实机测试 |
| 美术资源 | 优先使用用户提供的原始资源，不轻易修改 |

---

# 4. 长期约束

- 必须采用 Markdown 文档
- 必须文档职责唯一
- 必须支持 Context Recovery
- 不得删除用户提供的原始资源
- 不得将 art-resources 中像素风资源直接混进水墨风项目
- 不得将用户举例的配方（如 3水+2火）直接写成正式规则
- 当前目录不是 Git 仓库，操作需谨慎
