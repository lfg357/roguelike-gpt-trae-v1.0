# AI-POS 完整验证协议（手动触发）

> 使用方式：在 Trae 对话中输入 `#Rule ai-pos-verify` 手动激活。
> 触发场景：写入项目文档前、任务交付前、发现 AI 疑似编造时。

---

## 写入前验证（4 项检查）

每次向项目文档写入内容前：

| # | 检查项 | 通过标准 |
|---|--------|----------|
| 1 | 来源检查 | 内容来源：用户要求 / 项目文档 / 搜索结果 / 代码执行。否则停写 |
| 2 | 冲突检查 | 检查 DECISIONS 和 MEMORY，不违反已有决策和规则 |
| 3 | 职责检查 | 写入的文档是否正确归属？（参考 INDEX 信息归属索引） |
| 4 | 确认检查 | 用户是否确认？否则仅写草稿，标记 ⚠️ 待确认 |

---

## 任务完成验证

### 功能验证
```
□ 任务目标是否全部完成？
□ 是否有遗漏的子任务？
```

### 一致性验证
```
□ CURRENT / TODO / CHANGELOG 三者状态一致？
□ 当前任务 ID 是否在三处一致？
□ Recovery Snapshot 是否最新？
```

### 来源验证
```
□ 所有关键结论有明确来源？
□ 是否有 ⚠️ 或 ❓ 内容被写入文档？
□ 新 CHANGELOG 条目可追溯？
```

---

## 跨文件一致性检查

| 数据 | 应出现在 | 一致性要求 |
|------|----------|------------|
| 项目状态 | CURRENT、PROJECT | 必须一致 |
| 当前任务 ID | CURRENT、TODO | 必须一致 |
| 当前阶段 | CURRENT | 唯一 |
| 任务完成 | TODO → CHANGELOG | 完成的任务必须同步迁移 |
| Decision 编号 | DECISIONS | 自增，不重复 |

---

## 编造检测（5 问）

1. 这个结论能追溯到文档/搜索/用户吗？
2. 这个决策有用户确认记录吗？
3. 这个状态变更与 CHANGELOG 匹配吗？
4. 这个规则是用户明确要求的，还是 AI 自创的？
5. 这个 Decision 日期在实施之前吗？

---

## 人机确认协议

以下操作**必须**经人类确认：

| 操作 | 确认方式 |
|------|----------|
| 删除项目文件 | 用户明确口头授权 |
| 修改 DECISIONS | 用户确认方案选择 |
| 新增 MEMORY 长期规则 | 用户确认规则内容 |
| 改变项目状态 | 用户确认阶段变更 |
| 修改 PROJECT 目标 | 用户明确要求 |
| 对外发布/发送 | 用户确认交付物 |

---

## 验证失败处理

```
发现问题 → 分类（功能/一致性/来源/编造）→ 记录 → 修正 → 重新验证 → 通过后交付
```

严重问题（编造、多文件不一致、违反长期规则）→ 立即停止，告知用户。
