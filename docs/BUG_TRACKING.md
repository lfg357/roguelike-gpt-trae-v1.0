# 缺陷跟踪流程

> 定义缺陷的生命周期、分级标准、提单模板与流转规则  
> 版本：v1.0 | 日期：2026-07-14

---

## 1. 缺陷优先级分级

| 等级 | 定义 | 响应时间 | 修复时限 | 示例 |
|------|------|----------|----------|------|
| **S** | 致命：崩溃/死机/数据丢失/完全无法继续 | 立即 | 2h | 游戏启动崩溃、Boss战死机、层间切换卡死、存档损坏 |
| **A** | 严重：核心功能阻塞/主要流程中断 | 4h | 24h | 无法攻击、商店无法购买、五行盘无法操作、死亡后无法重开 |
| **B** | 中等：功能异常/体验明显受损 | 24h | 3天 | 格局计算错误、遗物效果不触发、敌人AI行为异常、伤害数值错误 |
| **C** | 轻微：UI错位/文字错误/次要功能异常 | 3天 | 1周 | 血条显示偏移、描述文字错误、粒子效果缺失、按钮点击区域偏小 |
| **D** | 建议：优化建议/非缺陷问题 | 排期 | 按需 | 动画帧率建议、颜色微调、交互优化提示、数值平衡建议 |

### 1.1 升级规则

- C级缺陷若影响核心流程，可升级为B级
- B级缺陷若导致流程中断，可升级为A级
- A级缺陷若导致崩溃/数据丢失，可升级为S级
- 升级须记录原因和升级人

## 2. 缺陷生命周期

```
[新建] → [确认] → [分配] → [修复中] → [待验证] → [关闭]
   ↑                                         ↓
   └──────────────── [拒绝/重复/无法复现] ←───┘
```

| 状态 | 说明 | 责任人 |
|------|------|--------|
| **新建** | 缺陷刚发现，尚未确认 | 发现人 |
| **确认** | 确认是缺陷，非误报 | 测试负责人 |
| **分配** | 指派给开发人员 | 测试负责人 |
| **修复中** | 开发人员正在修复 | 开发人员 |
| **待验证** | 修复完成，等待验证 | 开发人员 |
| **关闭** | 验证通过，缺陷关闭 | 测试负责人 |
| **拒绝** | 确认不是缺陷，拒绝 | 测试负责人 |
| **重复** | 与已有缺陷重复 | 测试负责人 |
| **无法复现** | 无法再次触发 | 测试负责人 |

### 2.1 状态流转规则

- 新建 → 确认：测试负责人确认缺陷真实存在
- 确认 → 分配：根据模块分配给对应开发人员
- 分配 → 修复中：开发人员开始处理
- 修复中 → 待验证：开发人员提交修复
- 待验证 → 关闭：测试人员验证通过
- 待验证 → 修复中：验证不通过，打回重修
- 新建 → 拒绝/重复/无法复现：测试负责人判定

## 3. 缺陷提单模板

```markdown
## 缺陷报告

### 基本信息
- **缺陷编号**：BUG-YYYY-NNN（如 BUG-2026-001）
- **发现日期**：YYYY-MM-DD
- **发现人**：
- **优先级**：S / A / B / C / D
- **状态**：新建
- **所属模块**：战斗 / 五行 / 遗物 / 层间切换 / 动画 / UI / 其他
- **关联用例**：TC-XXX（如有）

### 环境信息
- **Godot版本**：4.7
- **代码版本**：commit hash 或日期
- **操作系统**：Windows
- **运行方式**：实机 / headless

### 缺陷描述
【一句话描述问题】

### 复现步骤
1. 步骤一
2. 步骤二
3. 步骤三

### 期望结果
【应该发生什么】

### 实际结果
【实际发生了什么】

### 复现频率
[ ] 必现  [ ] 高（>50%）  [ ] 中（10%-50%）  [ ] 低（<10%）

### 附件
- [ ] 截图
- [ ] 录屏
- [ ] 日志文件
- [ ] 存档文件
- [ ] 其他：

### 备注
【其他补充信息】
```

## 5. 缺陷统计指标

| 指标 | 计算方式 | 目标 |
|------|----------|------|
| 缺陷发现率 | 发现缺陷数 / 测试用例数 | 参考值，无绝对目标 |
| 缺陷修复率 | 已关闭缺陷 / 总缺陷 | P0阶段>=90%，P1>=95% |
| 平均修复时长 | 总修复时长 / 关闭缺陷数 | S级<2h，A级<24h，B级<3天 |
| 复测失败率 | 复测失败次数 / 待验证缺陷数 | <10% |
| 遗漏缺陷率 | 发布后发现的缺陷 / 总缺陷 | <5% |

## 6. 缺陷存放位置

由于暂无缺陷管理系统，采用文件方式跟踪：

- **活跃缺陷**：`tests/bugs/ACTIVE/` 目录下，每缺陷一个 `.md` 文件
- **已关闭缺陷**：`tests/bugs/CLOSED/` 目录下
- **统计汇总**：`tests/bugs/BUG_STATISTICS.md`

### 目录结构
```
tests/bugs/
├── ACTIVE/
│   ├── BUG-2026-001.md
│   └── BUG-2026-002.md
├── CLOSED/
│   └── BUG-2026-000.md
└── BUG_STATISTICS.md
```
