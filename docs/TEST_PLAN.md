## 0. 项目架构分析

### 0.1 架构概览

### 0.3 数据流

```
输入层（Input）
    ↓
Player.gd（处理输入 → 更新玩家状态）
    ↓
Main.gd（同步玩家状态 → 广播信号）
    ↓
Enemy.gd / Room.gd / Relic.gd（响应信号 → 更新对应系统）
    ↓
FiveElementRules.gd（计算格局/组合 → 返回效果）
    ↓
Main.gd（应用效果 → 更新玩家属性）
    ↓
CombatFeedback.gd（播放反馈 → 视觉表现）
    ↓
渲染层（CharacterSequences / VFXSequences）
```

## 1. 测试目标

建立覆盖 P0 / P1 / P2 三阶段的完整测试体系，确保：
1. 核心功能（战斗、五行、遗物、层间切换）符合设计文档规格
2. 视觉表现（打击感、动画流畅度、VFX）达到验收标准
3. 数值平衡（DPS/EHP、敌人难度曲线、经济系统）在合理区间
4. 回归测试保障新功能不破坏已有系统

## 3. 测试策略

### 3.1 测试阶段

```
阶段1：冒烟测试（每次构建后自动执行）
  └─ 目的：验证项目可运行、核心流程不崩
  └─ 触发：代码变更后 / 每日构建
  └─ 执行：Godot headless 自动跑测
  └─ 输出：通过 / 失败 + 错误日志

阶段2：核心功能测试（功能开发完成后执行）
  └─ 目的：验证各系统符合设计规格
  └─ 触发：功能开发完成 / Bug修复后
  └─ 执行：Godot headless + 实机验证
  └─ 输出：测试报告 + 缺陷清单

阶段3：回归测试（每次大版本发布前执行）
  └─ 目的：验证新功能未破坏旧功能
  └─ 触发：P0/P1/P2 里程碑前
  └─ 执行：全量用例重新跑测
  └─ 输出：回归测试报告

阶段4：实机视觉验收（美术/表现变更后执行）
  └─ 目的：验证视觉表现达标
  └─ 触发：动画/VFX/角色变更后
  └─ 执行：实机运行 + 人工目视检查
  └─ 输出：验收清单签字
```

### 3.2 测试类型配比

| 类型 | 占比 | 执行方式 |
|------|------|----------|
| 自动化测试（冒烟 + 数据校验） | 40% | Godot headless 脚本 |
| 功能测试（用例执行） | 35% | 实机手动执行 |
| 视觉验收 | 15% | 实机人工检查 |
| 数值平衡验证 | 10% | 数据计算 + 实机体验 |

## 5. 缺陷优先级分级

| 等级 | 定义 | 响应时间 | 示例 |
|------|------|----------|------|
| **S** | 致命：崩溃/死机/数据丢失/完全无法继续 | 立即修复 | 游戏启动崩溃、Boss战死机、存档损坏 |
| **A** | 严重：核心功能阻塞/主要流程中断 | 24h内修复 | 无法攻击、商店无法购买、层间切换卡死 |
| **B** | 中等：功能异常/体验明显受损 | 3天内修复 | 格局计算错误、遗物效果不触发、敌人AI异常 |
| **C** | 轻微：UI错位/文字错误/次要功能异常 | 1周内修复 | 血条显示偏移、描述文字错误、粒子效果缺失 |
| **D** | 建议：优化建议/非缺陷问题 | 排期处理 | 动画帧率建议、颜色微调、交互优化 |

## 9. QA 测试计划执行规范

### 9.1 日常测试流程

```
开发完成 → 冒烟测试回归 → 核心功能测试 → 视觉验收（如涉及）→ 文档更新 → 标记完成
     ↓
  若失败 → 修复 → 重新测试
```

### 9.2 测试触发条件

| 触发场景 | 必须执行 | 建议执行 |
|----------|----------|----------|
| 代码修改（任何文件） | 冒烟测试 | 相关核心用例 |
| 新增功能 | 冒烟测试 + 核心用例 + 回归测试 | 视觉验收 |
| 美术资源变更 | 动画深度测试 | 实机视觉验收 |
| 数值调整 | 核心用例（数值相关） | 实机体验验证 |
| Bug修复 | 冒烟测试 + 相关用例 | 回归测试 |
